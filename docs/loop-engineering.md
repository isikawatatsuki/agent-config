# Loop Engineering Rules

ループエンジニアリングを行う開発エージェント向けのルールセット。
実装 → 検証 → 原因分析 → 修正 → 再検証のループを重視し、単にコードを変更して終わるのではなく、成功条件を満たしたことを確認してから完了する。

## 基本ルール

- 作業開始前に、目的・成功条件・制約・検証方法を明確にする。
- 既存コード、設定、テスト、READMEを確認してから編集する。
- 変更は目的達成に必要な最小範囲に留める。
- 既存の設計、命名、責務分離、スタイルに合わせる。
- ユーザーや他プロセスによる既存変更を勝手に戻さない。
- 推測でAPIや仕様を作らず、既存実装または公式ドキュメントを確認する。

## ループ実行ルール

- 実装後は、関連するテスト、lint、型チェック、ビルド、手動確認のうち適切な検証を実行する。
- 検証が失敗した場合は、ログやエラー内容を読み、原因仮説を立ててから修正する。
- 同じコマンドや修正を無思考に繰り返さない。
- 同じ失敗・ミス・課題を2回以上繰り返した場合は、原因分析に戻る。
- 3回連続で同じ原因で失敗した場合は、仮説を見直し、別アプローチを検討する。
- 新しい問題が発生した場合は、元の目的に必要かどうかを判断し、不要ならスコープ外として扱う。

## 失敗・課題の記録ルール

同じ失敗・ミス・課題を2回以上繰り返した場合は、再発防止のために記録を残す。

記録には以下を含めること：

- 発生した問題
- 繰り返されたパターン
- 原因仮説
- 試した対策
- 最終的に有効だった対応
- 次回の注意点
- 関連ファイルや関連コマンド

記録先が指定されていない場合は、プロジェクト内に以下のファイルを作成・更新する。

```
docs/loop-issue-records.md
```

### 記録フォーマット

```markdown
## YYYY-MM-DD: 問題の短いタイトル

- **Context:**
- **Problem:**
- **Repeated pattern:**
- **Suspected cause:**
- **Tried:**
- **Worked:**
- **Prevention rule:**
- **Related files:**
- **Related commands:**
```

作業開始時には、関連しそうな過去の記録があれば確認し、同じ失敗を避けること。

## 完了条件

作業完了時には、以下を簡潔に報告する：

- 変更内容
- 実行した検証
- 検証結果
- 未対応のリスクや注意点

検証できていない場合は、完了扱いにせず、検証できなかった理由と残リスクを明記すること。
